# Prompt01：规则中台架构设计与数据模型

## 角色

你是高级架构师、高级测试平台工程师、AI 工程师。你正在修复正式生产系统，不允许用 Mock 代替真实能力。

## 背景

fix002 已经具备：

1. 项目初始化向导。
2. 双文件导入初始用例。
3. 用例驱动预扫。
4. 规则草案。
5. 批次执行。
6. 一键反馈证据包。

但当前规则更多是“草案”和“展示”，还没有成为执行器核心能力。fix003 要把规则中台做成平台的基础设施。

## 目标

设计并实现统一规则中台，支持：

1. 规则类型统一。
2. 规则作用域统一。
3. 规则版本管理。
4. 规则审核流程。
5. 规则匹配与优先级。
6. 规则启用、停用、回滚。
7. 执行器可读取并使用规则。

## 规则类型

必须支持以下规则类型：

1. `login_rule`
   - 统一身份认证入口。
   - 用户名字段。
   - 密码字段。
   - 登录按钮。
   - 登录成功/失败证据。

2. `navigation_rule`
   - 菜单路径。
   - 树节点。
   - 选中态。
   - 右侧区域变化。
   - 新窗口/新标签页。

3. `table_rule`
   - 表格定位。
   - 表头识别。
   - 查询入口。
   - 隐藏查询框展开。
   - 行打开方式。
   - 分页。
   - 批量循环。

4. `form_rule`
   - 字段别名。
   - 必填识别。
   - 控件类型。
   - 默认值。
   - 正例、反例、边界值生成。

5. `approval_rule`
   - 审批意见。
   - 追加/覆盖。
   - 下一处理人。
   - 提交/审批/同意/通过/办理按钮。
   - 审批成功证据。

6. `dialog_rule`
   - 确认框。
   - 选择弹窗。
   - 提示框。
   - 弹窗关闭和确认。

7. `assertion_rule`
   - 成功提示。
   - URL 变化。
   - 标题变化。
   - 列表状态变化。
   - 页面区域变化。

8. `recovery_rule`
   - 失败后的补救动作。
   - 重新定位。
   - 重新等待。
   - 切换 iframe/page。
   - 触发人工介入。

9. `risk_rule`
   - 删除、作废、审批、提交、发布等高风险动作的门禁。

## 规则作用域

规则必须支持以下作用域，优先级从高到低：

1. case：用例级。
2. page：页面级。
3. project：项目级。
4. system：系统级。
5. organization：企业级。
6. global：通用级。

## 数据模型要求

当前分支不要求兼容历史数据库结构。执行 `Prompt00` 完成数据库备份后，可以重新设计规则、规则草案、页面知识等核心表结构。  
优先目标是结构清晰、易维护、执行器易读取，不需要为了旧字段或旧 JSON 结构做兼容。

需要支持：

1. `scope_type`
2. `scope_id`
3. `page_fingerprint`
4. `url_pattern`
5. `rule_version`
6. `parent_rule_id`
7. `status`
8. `priority`
9. `match_config_json`
10. `action_config_json`
11. `success_criteria_json`
12. `risk_level`
13. `requires_human_confirmation`
14. `production_enabled`
15. `created_from`
16. `created_from_id`
17. `rollback_to_rule_id`

如现有表结构不合理，可以删除、重命名、拆分或重建。旧数据不迁移也可以，但必须保留数据库备份。

## 输出要求

1. 给出完整设计文档。
2. 给出后端模型和 Schema 修改方案。
3. 给出数据库备份、清空、重建策略。
4. 给出规则 JSON 示例。
5. 给出回滚策略。

## 验收标准

1. 规则可按类型、作用域、状态、优先级查询。
2. 规则版本可追溯。
3. 规则草案可以转正式规则。
4. 高风险规则不能直接自动生产启用。
5. 新数据结构比旧结构更简单、更适合非专业维护。
