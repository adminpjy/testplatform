# fix006 完整设计：智能维护体验与项目级报告平台

## 1. 背景问题

现有系统已经具备基础模块，但在企业生产使用中会暴露以下问题：

1. 资产列表无分页  
   规则库、页面知识库、失败样本、人工介入记录、运行记录、用例列表等数据量上来后，页面会慢、难查、难维护。

2. 失败样本只是展示  
   失败样本和人工介入记录不应该只是历史记录，而应该成为“智能分析、生成方案、验证修复、沉淀规则”的入口。

3. 规则和 Prompt 配置过于专业  
   当前大量依赖 JSON，且规则、知识、Prompt、用例、失败样本之间有关联。非专业维护人员难以理解和配置。

4. 报告粒度不对  
   当前报告更偏单次运行。企业需要项目级总览，再下钻到模块、用例、运行、步骤和截图证据。

## 2. 设计目标

fix006 要把平台从“研发调试工具”升级为“企业测试维护平台”：

1. 列表体验统一：
   - 分页。
   - 筛选。
   - 搜索。
   - 排序。
   - 状态统计。
   - 批量操作。

2. 失败处理智能化：
   - 失败自动分析。
   - LLM 生成用户可读根因。
   - LLM 生成修复方案。
   - 一键生成规则草案。
   - 一键 dry-run。
   - 一键验证。
   - 一键反馈维护。

3. 配置低门槛：
   - 规则配置表单化。
   - Prompt 配置模板化。
   - 页面知识可视化。
   - JSON 仅作为高级模式。
   - LLM 辅助生成、解释、检查配置。

4. 报告项目化：
   - 项目总览。
   - 批次趋势。
   - 模块覆盖。
   - 用例通过率。
   - 失败分类。
   - 自动修复效果。
   - 下钻至单用例、单运行、单步骤。

5. 非专业人员可操作：
   - 页面用业务语言。
   - 每个问题给出下一步动作。
   - 每个配置项有解释、示例、LLM 辅助。
   - 高风险操作有门禁。

## 2.1 数据策略

当前分支不要求兼容历史数据和旧数据库结构。  
执行 fix006 前必须备份 `data/aitp.db`，备份后允许：

1. 清空历史运行记录、失败样本、规则、页面知识、Prompt 配置和报告数据。
2. 删除不合理旧表。
3. 重建更清晰的数据模型。
4. 调整接口路径和返回结构。
5. 用结构化表单模型替代旧的复杂 JSON 配置。

设计优先级：

1. 新版本效果优先。
2. 非专业人员易维护优先。
3. 数据结构清晰优先。
4. 不为兼容旧数据牺牲体验和正确性。

## 3. 功能设计

### 3.1 通用分页与筛选

适用模块：

1. 规则库。
2. 规则草案。
3. 页面知识库。
4. 失败样本。
5. 失败分析。
6. 人工介入记录。
7. 修复应用记录。
8. 测试运行记录。
9. 功能测试用例。
10. 项目报告。
11. Prompt 配置。

后端统一返回：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0,
  "totalPages": 0,
  "hasNext": false,
  "hasPrev": false,
  "filters": {},
  "sort": {}
}
```

前端统一组件：

1. `PaginatedDataTable`
2. `FilterBar`
3. `StatusFacet`
4. `SavedView`

### 3.2 智能失败工作台

失败样本不是“记录列表”，而是“待处理问题队列”。

每个失败样本展示：

1. 用户可读摘要。
2. 失败分类。
3. 影响用例和项目。
4. 关键截图。
5. 已尝试策略。
6. LLM 分析状态。
7. 推荐动作。
8. 修复状态。
9. 验证状态。

可执行动作：

1. 智能分析。
2. 生成修复方案。
3. 生成规则草案。
4. 生成人工介入方案。
5. dry-run。
6. 验证重跑。
7. 标记为系统缺陷。
8. 一键反馈。
9. 关闭问题。

### 3.3 人工介入记录升级

人工介入记录不是展示历史，而是学习来源。

处理流程：

1. 查看介入前页面。
2. 查看人工输入的处理步骤。
3. 查看介入执行结果。
4. LLM 总结可复用规则。
5. 生成规则草案。
6. 验证规则。
7. 启用规则。

### 3.4 规则库低代码化

规则配置分三层：

1. 简单模式  
   适合非专业维护人员：
   - 规则名称。
   - 适用范围。
   - 页面/菜单/按钮/字段。
   - 要执行的动作。
   - 成功判断。
   - 风险等级。

2. 向导模式  
   根据规则类型一步步填写：
   - 登录规则向导。
   - 菜单导航规则向导。
   - 列表规则向导。
   - 表单规则向导。
   - 审批规则向导。
   - 断言规则向导。

3. 高级 JSON 模式  
   保留给研发和高级维护人员。

LLM 辅助：

1. 根据失败样本生成规则。
2. 根据截图/DOM 解释规则。
3. 检查规则是否缺字段。
4. 检查规则风险。
5. 生成测试样例。
6. 生成验证计划。

### 3.5 页面知识库可视化

页面知识库要从 JSON 数据升级为业务视图：

1. 页面名称。
2. URL 模式。
3. 页面类型。
4. 关键区域。
5. 表格。
6. 表单。
7. 按钮。
8. 菜单路径。
9. 审批区。
10. 成功证据。
11. 关联规则。
12. 关联失败样本。
13. 最近命中情况。

### 3.6 Prompt 配置产品化

Prompt 配置不要让用户直接面对大段模板。

设计为：

1. Prompt 场景：
   - 自然语言解析。
   - DSL 生成。
   - 失败分析。
   - 规则生成。
   - 人工介入方案。
   - 测试数据生成。
   - 报告总结。

2. Prompt 版本：
   - 当前版本。
   - 草稿版本。
   - 历史版本。
   - 回滚。

3. Prompt 变量：
   - 变量说明。
   - 示例值。
   - 是否必填。
   - 脱敏策略。

4. LLM 辅助：
   - 解释当前 Prompt。
   - 优化 Prompt。
   - 检查变量缺失。
   - 用样本试运行。
   - 对比新旧输出。

### 3.7 项目级报告

报告入口按项目组织。

项目报告总览：

1. 项目基本信息。
2. 测试范围。
3. 用例总数。
4. 已执行数。
5. 通过数。
6. 失败数。
7. 阻塞数。
8. 通过率趋势。
9. 模块覆盖。
10. 风险操作统计。
11. 自动修复次数。
12. 人工介入次数。
13. 未解决问题。

下钻层级：

1. 项目。
2. 测试活动/批次。
3. 模块。
4. 功能点。
5. 用例。
6. 运行记录。
7. 步骤。
8. 截图/日志/DOM/trace。

导出：

1. HTML。
2. Word/PDF。
3. Excel 明细。

## 4. 权限与安全

角色建议：

1. 测试人员：
   - 查看报告。
   - 执行用例。
   - 提交反馈。

2. 测试负责人：
   - 创建批次。
   - 审核报告。
   - 分配失败问题。

3. 维护人员：
   - 处理失败样本。
   - 生成规则草案。
   - dry-run。

4. 管理员：
   - 启用高风险规则。
   - 管理 Prompt。
   - 配置 LLM。
