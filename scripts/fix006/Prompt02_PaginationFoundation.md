# Prompt02：通用分页、筛选、排序基础设施

## 目标

为所有高增长数据模块增加分页能力，避免页面加载慢、数据难查、维护困难。

## 后端设计

新增统一分页 Schema：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0,
  "totalPages": 0,
  "hasNext": false,
  "hasPrev": false
}
```

支持查询参数：

1. `page`
2. `page_size`
3. `q`
4. `status`
5. `type`
6. `project_id`
7. `case_id`
8. `run_id`
9. `sort_by`
10. `sort_order`
11. `created_from`
12. `created_to`

## 需要分页的接口

至少覆盖：

1. 规则库。
2. 规则草案。
3. 页面知识库。
4. 失败样本。
5. 失败分析。
6. 人工介入记录。
7. 修复应用记录。
8. 测试运行记录。
9. 项目用例。
10. Prompt 配置列表。

## 前端设计

新增通用组件：

1. `PaginatedDataTable`
2. `PaginationBar`
3. `FilterBar`
4. `StatusFacet`
5. `SavedViewSelector`

前端要求：

1. 默认每页 20 条。
2. 支持 10、20、50、100。
3. 支持关键词搜索。
4. 支持状态筛选。
5. 支持 URL query 保留筛选条件。
6. 切换页不丢筛选条件。

## 重构要求

当前分支不要求兼容旧接口。  
可以直接把原列表接口升级为分页结构，也可以重命名接口，只要前端同步修改并通过测试。

设计优先：

1. 接口统一。
2. 前端调用简单。
3. 大数据量性能稳定。
4. 不为保留旧返回结构牺牲新体验。

## 验收标准

1. 规则库分页可用。
2. 页面知识库分页可用。
3. 失败样本分页可用。
4. 人工介入记录分页可用。
5. 数据量 1000 条时页面仍可操作。
6. 旧的无分页列表可以被替换，不需要兼容保留。
