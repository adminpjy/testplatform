# fix002 企业级 AI 功能测试平台设计方案

## 目标

本次修复不再围绕某一个页面或某一种控件打补丁，而是把系统升级为“用例驱动、规则优先、LLM 兜底、失败可反馈、批量可报告”的生产级测试平台。

核心目标：

1. 简化项目初始化：用向导完成项目配置、资料上传、初始用例生成、账号配置、预扫增强。
2. 接入前置程序：支持上传项目必备的两个文件，调用或模拟前置比对结果导入初始功能测试用例。
3. 以初始用例为中心预扫：不做高风险的全系统自由探索，只围绕用例路径识别页面、列表、表单、审批面板、成功证据。
4. 形成规则沉淀：将预扫结果、失败分析结果沉淀为可审阅、可启用的规则草案。
5. 支持批量测试：一个项目可以执行上百上千条用例，形成测试活动级报告，而不是每条用例孤立成一个报告。
6. 失败自愈闭环：失败后自动打包日志、截图、DSL、DOM、运行流、异常栈，生成 LLM 分析提示词，产出规则草案或用例修复建议，可预执行验证。
7. 一键反馈：无法自动解决时，生成完整后台维护工单，维护人员可以直接复盘完整路径和证据。

## 非目标

1. 暂不做“理解整个系统”的无边界探索，避免复杂度和误操作风险。
2. 不在生产环境自动修改代码。系统可自动生成规则草案、用例修复草案、人工介入方案，但代码级升级必须由维护人员审核。
3. 不用 Mock 替代真实系统行为。测试执行仍基于真实页面、真实截图、真实 DOM 和真实运行记录。

## 总体架构

### 分层

1. 项目初始化层
   - 项目基本信息、系统 URL、登录方式、账号权限、LLM 服务、运行安全策略。
   - 上传两个项目必备文件。
   - 调用前置比对程序或解析前置输出，生成初始功能测试用例。

2. 用例资产层
   - 功能测试用例、版本、DSL、测试数据、前置条件、成功标准。
   - 用例来源包括：前置程序、文档提取、手工创建、失败重放、AI 增强。

3. 预扫增强层
   - 按用例路径预扫，不做全站扫描。
   - 识别菜单路径、查询条件、列表、行打开方式、新页面、表单字段、审批意见、提交按钮、成功提示。
   - 生成 page_profile、form_rule、table_rule、approval_rule、navigation_rule 等规则草案。

4. 执行层
   - 规则优先：明确 DSL、项目规则、系统规则、通用规则。
   - LLM 辅助：规则不够时使用 DOM/可访问性树/截图做判断。
   - 视觉兜底：多模态模型识别页面元素并返回坐标或候选控件。
   - 人工介入：LLM 生成人工介入方案，一键执行后生成可沉淀规则。

5. 批次与报告层
   - 测试活动、测试批次、批次用例、运行记录、缺陷候选、规则修复记录。
   - 输出系统级报告：覆盖范围、通过率、失败分类、自动修复、人工介入、风险操作、未解决问题。

6. 反馈与维护层
   - 一键反馈生成维护工单。
   - 附带完整路径：项目、用例、DSL、运行、步骤、截图、过程截图、DOM、可访问性树、Runtime Stream、locator-debug、execution trace、LLM 调用摘要。
   - 支持维护人员从工单转规则、转缺陷、转用例修复。

## 数据模型

### 新增模型

1. `ProjectBootstrapPackage`
   - 记录项目初始化包，两份输入文件、前置程序输出、导入摘要、状态。

2. `TestCampaign`
   - 表示一次完整测试活动或批次。
   - 统计总用例、已执行、通过、失败、阻塞、自动修复次数。

3. `TestCampaignCase`
   - 记录批次中的每条用例及对应 run_id、执行状态、失败摘要。

4. `PrescanSession`
   - 记录用例驱动预扫任务。
   - 产出页面知识、规则草案、增强后的用例建议。

5. `MaintenanceFeedback`
   - 一键反馈工单。
   - 保存完整证据包路径、摘要、状态、维护处理建议。

## 规则体系

规则优先级：

1. 用例 DSL 明确指令。
2. 项目级规则。
3. 被测系统级规则。
4. 通用能力规则。
5. LLM 候选判断。
6. 多模态视觉兜底。
7. 人工介入。

规则类型：

1. `navigation_rule`：菜单路径、树节点、面包屑、选中态。
2. `table_rule`：列表定位、行打开方式、分页、隐藏查询框。
3. `form_rule`：字段别名、必填项、输入格式、默认值。
4. `approval_rule`：意见框、追加/覆盖、下一处理人、提交/同意/审批按钮。
5. `dialog_rule`：确认框、提示框、选择弹窗。
6. `assertion_rule`：成功标识、页面变化、URL、新页面、状态提示。
7. `recovery_rule`：失败后的恢复动作。
8. `risk_rule`：删除、作废、提交、审批、发布等高风险操作控制。

## 项目初始化向导

向导步骤：

1. 项目基础信息：项目名称、系统名称、base_url、login_url、home_url、认证方式。
2. 账号与权限：默认测试账号、读写审批删除权限。
3. 上传资料：上传项目必备两个文件，并记录文件角色。
4. 生成初始用例：调用前置程序输出或用 LLM/规则解析文本生成初始用例草案。
5. 导入用例：预览、编辑、批量接受为正式用例。
6. 预扫增强：选择用例范围，执行只读预扫，生成规则草案与 DSL 增强建议。
7. 批次执行：创建测试活动，按风险策略执行。

## 预扫策略

预扫只围绕初始用例，不做全系统漫游。

预扫采集：

1. 页面标题、URL、iframe、新页面行为。
2. 菜单路径节点与选中态。
3. 表格容器、表头、行打开方式、查询入口。
4. 表单字段、字段别名、控件类型、必填标识、校验提示。
5. 审批区：意见框、意见写入方式、下一处理人、提交类按钮。
6. 成功证据：toast、弹窗、列表回退、URL 变化、状态字段变化。

预扫产出：

1. 页面知识 `AbilityKnowledge`。
2. 规则草案 `RuleDraft`。
3. 用例增强建议：补充 DSL、测试数据、成功标准。
4. 预扫报告。

## 失败自愈闭环

流程：

1. 失败发生后，执行器必须保存每一步过程截图和完整日志。
2. 后端生成 failure sample。
3. 自动构建 evidence package。
4. 调用 LLM 分析根因，生成：
   - 用户可理解错误说明。
   - 可执行修复建议。
   - 规则草案。
   - DSL 修复补丁。
   - 人工介入方案。
5. 对低风险规则执行 dry-run 或 replay 验证。
6. 验证通过后进入待启用规则草案。
7. 验证失败或高风险时生成一键反馈工单。

## 批次报告

报告层级：

1. 测试活动。
2. 模块。
3. 功能点。
4. 用例。
5. 步骤。

报告内容：

1. 总体通过率、失败率、阻塞数。
2. 正例、反例、边界、异常场景统计。
3. 模块覆盖与功能覆盖。
4. 失败分类：定位失败、登录失败、数据缺失、权限问题、页面变更、系统缺陷。
5. 自动生成规则与已验证规则。
6. 人工介入次数和效果。
7. 高风险操作审计。
8. 未解决问题和维护反馈链接。

## 生产安全

1. 查询、查看、导出：可自动继续。
2. 新增、修改、上传：需要明确测试数据和成功标准。
3. 删除、作废、审批、提交、发布：必须满足账号权限、目标明确、规则命中、成功证据清晰，否则停止并提示。
4. 未知失败不能静默跳过，必须生成可读原因和反馈入口。

## 本次 fix002 实施范围

第一期落地以下能力：

1. 后端新增项目初始化向导 API，支持双文件初始化包、初始用例草案生成、批量导入。
2. 后端新增用例预扫会话 API，产出规则草案和用例增强建议。
3. 后端新增测试批次 API，支持按项目选择用例并批量创建运行记录，维护批次状态和报告摘要。
4. 后端新增一键反馈 API，基于运行记录或失败样本打包完整证据。
5. 前端新增“项目向导”入口，简化项目创建、资料上传、用例导入、预扫、批次执行。
6. 补充单元测试覆盖导入、预扫、批次、反馈包。
